# ADR-0001: AIネイティブなシステム構築専門技術スタジオとしてのポジショニング定義とクリーンアップ

- **Date**: 2026-06-09
- **Status**: Accepted
- **Deciders**: ayato-labs (Human), Antigravity (AI Agent)

## Context

Ayato Studio Portal は、当初AIニュースアグリゲーション、LLMスペック統計、学習用アカデミー、個人技術ブログ、および内製アプリケーション（OAuth認証機能）といった極めて多様なコンテンツや機能を1つのドメイン内で並行して提供していました。

しかし、この構成には以下の問題が存在していました：
1. **ユーザーフローの分散**: 訪問者が「ニュースを読みに来たビジネスパーソン」なのか「実装技術を追うエンジニア」なのか「開発案件を依頼したい企業担当者」なのかによって、とるべきアクション（コンバージョンパス）が衝突していました。
2. **E-E-A-T（SEOにおける専門性）の希薄化**: 数学学習（academy）や個人エッセイ、実務技術などが混在し、検索エンジンに対してドメインの専門性が定義しにくい状態でした。
3. **デッドコードの存在**: 認証機能（auth）を要件定義上でNon-Goalとしていながら、関連するAPIや callback 処理のコードがプロジェクト内に残存しており、セキュリティリスクとエッジランタイムでのオーバーヘッドを生んでいました。

### 競合分析と差別化戦略
日本のAI開発・システム構築市場における各プレイヤー（競合）の特性と、Ayato Studioのポジショニングを以下のように分析しました。

| 競合セグメント | 主なプレイヤー・特徴 | ターゲット・顧客のペインポイント（弱点） | Ayato Studioの差別化戦略 |
| :--- | :--- | :--- | :--- |
| **SIer（大手・中堅）** | NTTデータ、NRI、TIS、クラウドエース、ID社など。MCP導入支援や大規模エージェント開発に参入開始。 | ・初期費用や運用保守費が極めて高額（数百万〜数千万円）<br>・ウォーターフォール前提で意思決定と検証スピードが遅い<br>・自社製クラウドインフラ依存によるベンダーロックインや高額な月額固定費。 | **超高機動型エッジファースト**<br>・Cloudflare Pages + Supabase RLS等を駆使し、インフラ維持費ほぼゼロ（サーバーレス）での最速PoC・MVP開発。<br>・アジャイルかつ数週間でのリリースサイクル。 |
| **SaaS（AIエージェント・RAG構築）** | Dify、Coze、miibo、Kore.aiなど。GUIによる手軽なAIエージェント構築を提供。 | ・複雑な社内レガシーシステムや特殊プロトコル（独自のローカルDBなど）への連携が困難。<br>・API通信や従量課金によるランニングコスト高騰。<br>・機密データがサードパーティSaaSを経由することによるガバナンスリスク。 | **カスタムMCPサーバー＆コードファースト連携**<br>・標準仕様MCP（Model Context Protocol）に準拠したセキュアなカスタムMCPサーバーを個別に実装。<br>・SaaSでは実現できない、既存社内システムやオンプレミスDBとのセレクト連携を実現。 |
| **SES（常駐派遣）** | AIエンジニア、プロンプトエンジニアのマンパワー派遣会社。 | ・要件定義やエージェントハーネス設計を自律的に推進できる高スキル人材の不足。<br>・派遣メンバーの個人スキルに依存し、開発ノウハウが顧客企業内に蓄積されない。 | **スタジオ型ソリューション提供**<br>・単なる要件沿いの「人月貸し」ではなく、エージェントインフラ設計（LLMメモリ管理、ガードレール実装など）をスタジオ発の完成度高いアーキテクチャとして直接納品。 |
| **X・個人開発者・インフルエンサー** | AIインフルエンサー、小規模フリーランスチーム。新技術のデモやCursor/Claudeのハウツーを発信。 | ・検証目的（PoC）で終わることが多く、プロダクション品質（セキュリティ、エラー耐性、監査ログ、DB整合性）の担保が不十分。<br>・長期的な保守・バージョンアップ体制への懸念。 | **オープンナレッジによる信頼と本番クオリティの担保**<br>・「Tilde Research風」に、MCPのSQLite競合バグやガードレール実装など、ディープな技術トラブルシューティングをInsightsで無償公開。<br>・オープンソース貢献や高度なインサイト発信により信頼性を担保し、品質で差別化。 |

このように、従来の「AIコンサル」「プロンプト作成教育」といったレッドオーシャンから脱却し、**「AIが存在することを前提とした、高機能で維持費ゼロのAIネイティブシステム・インフラを最速で構築できる技術スタジオ」**という独自のポジション（ブルーオーシャン）を確立します。

そのため、ブランドポジショニングを「AIネイティブなシステム構築」へ先鋭化し、不要な機能を削ぎ落としたクリーンな設計へ整理整頓（リファクタリング）する必要が生じました。

## Decision

以下の通り、ブランドポジショニングの再定義とアーキテクチャ・ルーティングの整理整頓を決定しました。

1. **ブランドポジショニングの定義**:
   - サイトの目的とブランドを「**AIが存在することを前提とした、AIネイティブなシステム・ワークフロー構築の専門技術スタジオ**」に特化させ、BtoBの受託開発・インテグレーション案件の獲得を最終コンバージョン（CV）とします。AIエージェントに心地よく働いてもらうためのインフラ技術（MCP、メモリ管理等）と、AI APIをアプリケーションロジックに深く組み込む技術（ハーネス・エンジニアリング）を中核アセットとします。

2. **ルーティングおよびコンテンツの削減（引き算）**:
   - `auth`（認証関連）を完全に削除します。
   - `academy`（学習コンテンツ）を廃止します。これにより、ドメインの専門性を「初学者向け教育」から「高度な実務インテグレーション」へシフトさせます。

3. **フロー型とストック型のUI再編成**:
   - 習慣的訪問者（ファン）を構築するため、トップページのファーストビュー最上部にフロー型の「AI News Bulletin Board」（自動生成ニュース）を配置します。
   - ストック型の技術ブログを `blog` から `insights` へ改名・移動し、トーン＆マナーを「AIエージェントインフラ・ハーネスエンジニアリング」に特化した内容へと書き換えます。

4. **アプリ実績の簡素化**:
   - `/apps`（Gmailプロテクター等）は、Google API等の外部審査に必要なプライバシーポリシーおよび利用規約のURL（ルーティング）は維持しつつ、ユーザー画面は「GitHubリンク＋Markdownベースの技術資料」へと統一・簡素化します。これにより自作ツール群を「スタジオの開発力証明（アセット実績）」として位置づけます。

## Consequences

### Positive
- **CVの最大化**: 技術力を証明するReportsやInsightsから、サービス（Services）および問い合わせ（Contact）へのスムーズな回遊フローが一本化され、BtoBの案件相談へ誘導しやすくなります。
- **SEO（E-E-A-T）の向上**: ドメインのトピッククラスターが「AIエージェント開発と統合技術」に絞り込まれ、検索インデックスの専門性が強まります。
- **パフォーマンスとセキュリティ向上**: `/auth` や不要なライブラリ、不要なルーティングが物理的に削除され、エッジ環境（Cloudflare Pages/Workers）での起動時間の短縮および脆弱性アタックサーフェスが減少します。

### Negative / Risks
- **URL変更に伴う影響**:
  - `/blog/[slug]` から `/insights/[slug]` への変更、および `/academy` の削除により、古いリンクへのアクセスが404エラーになるリスクがあります。
  - **対策**: Cloudflare Pagesの `_redirects` 設定を用いて、`/blog/*` から `/insights/:splat` への 301 Redirect ルールを記述し、検索インデックスおよび被リンクを保護します。
