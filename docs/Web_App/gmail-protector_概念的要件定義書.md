# Gmail Protector 概念的要件定義書

## 1. プロダクトの目的とコンセプト
**「AIによる送信前リアルタイム監査で、機密漏洩のヒューマンエラーをゼロにする」**
Gmail上でのメール送信直前に、Gemini APIを用いて文面や添付ファイルを自動解析し、意図せぬ情報漏洩や不適切なトーンを警告するChrome拡張機能。

## 2. 実装する機能と方針 (In-Scope & Philosophy)

### クライアントサイドでのリアルタイムフック
*   **What**: Chrome Extension の Content Script を用い、Gmail の「送信ボタン」クリックイベントをフックして検証を挟む。
*   **Why**: ユーザーのワークフローを妨げず、かつ別のタブやアプリケーションを開くことなく、最も効果的なタイミング（送信直前）で監査を行うため。

### Bring Your Own Key (BYOK) モデルによる直接通信
*   **What**: ユーザー自身が取得した Gemini API キーを設定画面に入力し、ブラウザから直接 Google の API エンドポイントへリクエストを送信する。
*   **Why**: 自社サーバー（Ayato Studio側のバックエンド）を一切経由させないことで、メール本文という極めて機密性の高いデータを私たちが「保持しない・見ない」ことを構造的に保証する。同時に、サーバーの運用コストをゼロに抑える。

## 3. 実装しない機能と方針 (Non-Goals & Omissions)

### サーバーサイドでのメール一括スキャン（バッチ処理）
*   **What**: Gmail API (OAuth) を用いてバックエンドサーバーが定期的にユーザーのメールボックス全体をクロール・解析するアーキテクチャ。
*   **Why**: 実装しない。プライバシーリスクが極めて高く、Google の厳しいセキュリティ審査（CASA等）を通過するための莫大なコストと時間がかかるため。本ツールはあくまで「送信時のエンドポイント防御」に特化する。

### 強制的な「送信ブロック」機能
*   **What**: AIが危険と判定した場合、ユーザーの意志に関わらず送信をシステム的に「禁止（不可）」にする機能。
*   **Why**: 実装しない。LLMには常にハルシネーション（誤検知）のリスクが伴う。「AIはあくまで監査・警告を行うアドバイザーであり、最終的な送信の意思決定と責任は人間のユーザーにある」という哲学に基づき、警告を無視して送信する自由（Bypass機能）を必ず残す。

### 企業向けの一元管理ダッシュボード
*   **What**: 情シス部門が全社員のルール設定や検知ログを一元管理する SaaS 機能。
*   **Why**: 実装しない。本ツールは「個人の生産性と安全性を高めるパーソナル・アーマー」であり、B2B向けのコンプライアンス管理ツールではない。管理機能の開発オーバーヘッドを排除する。
