# Site Downloader 概念的要件定義書

## 1. プロダクトの目的とコンセプト
**「ブラウザの『見たまま』を、確実かつセキュアに手元へアーカイブする」**
動的にレンダリングされるモダンなSPAや、認証の壁の内側にあるコンテンツを、ユーザーのブラウザコンテキストから直接抽出し、オフライン閲覧可能な形でダウンロードするChrome拡張機能。

## 2. 実装する機能と方針 (In-Scope & Philosophy)

### ブラウザコンテキストでのDOM/アセット抽出
*   **What**: 外部のクローラーサーバーを使わず、拡張機能が現在開いているタブの DOM ツリーと関連アセット（画像、CSS等）を直接読み取り、ZIP化してダウンロードする。
*   **Why**: ログインが必要な会員サイトや、JavaScriptでクライアントサイドレンダリングされるページ（React/Vue等）を、外部サーバーからスクレイピングすることは困難または不可能である。ユーザーのブラウザが「レンダリングし終わった後の確実な状態」をキャプチャするため。

### GitHub Actions 経由の Chrome Web Store 自動デプロイ
*   **What**: リポジトリの `main` ブランチに変更がプッシュされた際、CI/CDパイプラインが自動的にビルドを行い、Chrome Web Store API を叩いてストアの下書き（Draft）を更新する。
*   **Why**: 拡張機能のデプロイは手作業（ZIP作成、ポータルへのアップロード）が多く、リリースサイクルのボトルネックになりやすい。これを完全自動化し、「労働からの解放（Liberation from Toil）」という Ayato Studio の理念を開発プロセス自体でも体現するため。

## 3. 実装しない機能と方針 (Non-Goals & Omissions)

### クラウドベースのスクレイピング・バックエンド
*   **What**: ユーザーがURLを指定し、バックエンドの Node.js (Puppeteer/Playwright) サーバーが代理でアクセスしてデータを取得する仕組み。
*   **Why**: 実装しない。Cloudflare等の高度なWAFにブロックされるリスクが高く、IPバンやサーバーの維持コストなどの運用負債を抱え込むため。「ローカルファースト」のアプローチを徹底する。

### クラウドストレージ（Google Drive / Dropbox）への直接同期
*   **What**: 抽出したサイトデータを、ローカルディスクではなく直接外部のクラウドストレージへ保存・同期する機能。
*   **Why**: 実装しない。OAuth連携やトークン管理など、認証周りの実装複雑度とメンテンナンスコストが跳ね上がる。ZIPファイルとして手元に落とし、それをどう管理するかは OS やユーザーの既存のファイル管理システムに委ねる（関心の分離）。

### DRM保護コンテンツやペイウォールの迂回ハック
*   **What**: 動画のDRM（Widevine等）を解除したり、有料記事の隠された部分を強制的に復号する機能。
*   **Why**: 実装しない。本ツールはあくまで「ユーザーが正当な権限を持って画面上に表示させている情報の個人的なアーカイブ」を目的としており、規約違反や違法行為を助長するスクレイパーではない。
