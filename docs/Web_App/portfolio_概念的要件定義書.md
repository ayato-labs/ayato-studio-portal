# Portfolio Strategist 概念的要件定義書

## 1. プロダクトの目的とコンセプト
**「売却による課税を避け、追加投資のみで理想のアセットアロケーションへ到達する」**
投資家が直面する「リバランス時の税金ロス」という課題に対し、ノーセル（売却なし）での資産比率調整を支援するシミュレーター。プロフェッショナルな資産形成を、最も軽量かつセキュアな形で提供する。

## 2. 実装する機能と方針 (In-Scope & Philosophy)

### Local Storage を用いた状態保存と計算
*   **What**: ユーザーが入力した資産額や目標比率をブラウザの Local Storage にのみ保存し、クライアント側で計算を完結させる。
*   **Why**: ユーザーの金融資産という極めてセンシティブな情報を扱うため、「サーバーにデータを一切送信しない」という構造的なプライバシー保護（Zero-Data Architecture）を実現する。また、バックエンド不要による「運用コストゼロ」を達成するため。

### VQE（価値定量化エンジン）の統合
*   **What**: ツール利用後に「有用性」や「節約できたリサーチ時間」をユーザーに問うフィードバックUIを組み込む。
*   **Why**: ツールが単なる便利機能で終わらず、「どれだけのビジネス価値（または時間の節約）を生み出したか」を定量的に計測し、ポータル全体の存在価値を証明するため。

## 3. 実装しない機能と方針 (Non-Goals & Omissions)

### リアルタイムな株価API連携
*   **What**: 外部API（Yahoo FinanceやAlpha Vantage等）から最新のティッカー価格を自動取得する機能。
*   **Why**: 実装しない。本ツールは「戦略的シミュレーション」が目的であり、日々の価格トラッキングポートフォリオツールではない。外部APIへの依存はシステムの脆弱性（レートリミット、API仕様変更、遅延）を生むため、あえて「手動入力によるスナップショット計算」に限定する。

### 「売却」を伴うリバランス計算
*   **What**: 基準比率を超過した資産を売却し、不足している資産を買うという一般的なリバランスの計算。
*   **Why**: 実装しない。本アプリのコアバリューは「ノーセル（非売却）による税制優遇の最大化」にある。売却を許容するUIは、アプリのコンセプト自体を自己否定することになるため意図的に排除する。

### ユーザーアカウント管理とクラウド保存
*   **What**: Supabase Auth 等を用いたログイン機能と、複数デバイス間でのポートフォリオデータの同期。
*   **Why**: 実装しない。シミュレーションは通常単一のデバイスで完結する作業であり、アカウント登録という摩擦（フリクション）は直感的なUXを著しく損なう。プライバシーと手軽さを最優先する。
