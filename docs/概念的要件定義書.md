# 要件定義(コンセプトドキュメント)
アプローチ: Supabase の Database Webhooks を使う(推奨🌟)
Supabase の機能で、「generated_reports テーブルに新しいデータが INSERT(保存)されたら、自動的に Cloudflare Pages の Deploy Hook URL にアクセスする」というトリガーを設定します。

Cloudflare ダッシュボードで環境変数を設定してください。
変数名: NEXT_PUBLIC_SUPABASE_URL と NEXT_PUBLIC_SUPABASE_ANON_KEY
設定場所: 「環境(ビルド)」と「環境(ランタイム)」の両方を「すべて」に設定して保存。

export const dynamicParams = true; に変更する(ビルド後に増えた ID でも動的に生成するようになります)。
または、export const revalidate = 60;(1分ごとに再検証)などの ISR 設定を追加する。

メリット: Reporter 側の Python コードには一切手を加える必要がなく、アーキテクチャとして非常にクリーンです。

posted_x と posted_blueskyってことは、XとBlueSkyに投稿した内容「どの記事をいつどこに投稿したか」という**実行状態(ステート)**については、保存しない

## 1. プロジェクトの目的と概要
Ayato Reporter が生成するレポートを、ユーザーが直感的かつ快適に閲覧できるフロントエンド・ポータルサイト(Ayato Studio Portal)を構築する。
「運用コストゼロ・手間ゼロ」を突き詰めるため、バックエンドにサーバーを持たないモダンなサーバーレスアーキテクチャ(Next.jsなどのSSG/SPA + Supabase + Cloudflare Pages)を採用する。

## 2. 実装する機能 (In-Scope Features)
ユーザー体験のコアとなる以下の機能に絞って実装する。

* **レポート一覧ページ (Feed)**:
    * Supabase に蓄積された最新のレポート一覧を取得し、タイムライン形式で表示。
    * カテゴリ(Technology, Finance, Energy など)に基づくシンプルな絞り込み表示。
* **レポート詳細ページ (Article)**:
    * Markdown 形式のレポート本文を、美しく読みやすいスタイルでレンダリング(シンタックスハイライト、見出しの構造化など)。
* **徹底したモバイルファーストUI**:
    * スマホでの閲覧を最優先としたレイアウト設計(レスポンシブデザイン)。親指で操作しやすいナビゲーションや、視認性の高いタイポグラフィの採用。

## 3. 実装しないと決めた機能 (Non-Goals)
システムをシンプルに保ち、メンテナンスコストと開発速度を優先するため、以下の機能は「あえて実装しない(捨てる)」と定義する。

* **ユーザー認証・ログイン機能 (Auth)**:
    * 現状は一般公開用ポータルとし、複雑なセッション管理やユーザー単位のパーソナライズは実装しない。
* **レポートの作成・編集機能 (CMS機能)**:
    * ポータルは純粋な **「閲覧専用 (Read-Only)」** アプリケーションとする。レポートの生成や修正はすべてバックエンドの AI Agent(`ayato_reporter`)の管轄とする。
* **Node.js サーバーに依存する動的レンダリング**:
    * 運用を複雑化させるコンピュートサーバー(完全なSSR)は使わず、エッジ配信が可能な静的サイト(Static/Edge)としてビルドする。
* **高度な全文検索エンジン**:
    * Algolia などの外部検索SaaSの導入は見送る。検索が必要な場合も Supabase の機能の範囲内に留める。

## 4. 制限事項とアーキテクチャ上の対応 (Limitations & Solutions)

### 4.1 デプロイメントの制限(セキュリティと手間のトレードオフ)
* **制限・課題**: CI側(GitHub Actions等)からデプロイを叩く構成にすると、GitHub側にCloudflareの強い権限を持つ API トークンや Account ID を機密情報として持たせる必要があり、管理リスクと手間が増える。
* **対応策(自律型デプロイ)**: 
  **「Cloudflare Pages 側が GitHub リポジトリ(の main ブランチなど)の変更を検知し、Cloudflare の基盤上で直接ソースを Pull してビルドする方式」** を採用する。これにより、コードベースやデプロイ環境に一切のデプロイ用機密情報を持たせずに済む。

### 4.2 サーバーレス(DB直接接続)の制限
* **制限・課題**: バックエンドサーバーを持たず、フロントエンドから直接 Supabase へアクセスするため、認証パラメーター(Anon Key など)がブラウザ上に露出する。
* **対応策(DB層での防御)**: 
  Supabase 側で **Row Level Security (RLS)** を厳格に設定する。「対象テーブルは誰でも Read (Select) できるが、Insert / Update / Delete は絶対にできない」というポリシーをDB層で敷くことで、クライアント側のキーを利用した悪意あるデータ改ざんを完全に無効化する。

### 4.3 マルチデバイス対応のリソース制限
* **制限・課題**: 開発リソースとスピードの観点から、PC、タブレット、スマホのすべてに対して100%完璧なカスタムデザインを提供することは非効率である。
* **対応策(スマホ体験への一点集中)**: 
  **UIはスマホでの閲覧に徹底的に最適化する。** ユーザーがスキマ時間に消費するユースケースを想定し、PCでの表示は「画面の中央にスマホ幅で表示される」などのフォールバック的な対応(ただ広がるのではなく、読める状態を維持する)にとどめ、PC特有の多段カラムレイアウトなどは構築しない。
